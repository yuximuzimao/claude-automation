## P0：资金安全问题（直接影响退款正确性）

### 问题 1：ERP 多行状态只读第一行 + "交易成功"未识别为已发货
**影响范围：6条差评**  
`fb-1776918894171` / `fb-1777099604107` / `fb-1777109171484` / `fb-1777140840406` / `fb-1777141379115` / `fb-1777099128076`

**根因：** `getErpStatus()` 只读 `rows[0].status`。对于分包订单（多行结果），后续行的状态被完全忽略。  
- "交易成功"（买家已完成支付/收货交易状态）未被识别为"已发货"，导致进入 fallback `escalate("ERP状态未识别")`
- 赠品有"交易成功"的分包但主商品是"待打印快递单"，系统认为赠品未发货 → 错误 approve 仅退款

**典型案例：**
- `100001776903448406557`：主商品待打印（未发货），赠品有"交易成功"分包快递已揽收 → 系统 approve（错误，需要拦截赠品）
- `100001777105523010216`：3行全部"交易成功"/"卖家已发货"→ 读第一行"交易成功"报状态未识别
- `100001777030582649173`、`100001777099317226286`：工单状态"用户-已取消" + ERP 也是"交易成功"→ 两层都识别错误

**修复方案：** 重写 `getErpStatus()` 为聚合版本，读取所有行，识别是否有任何行属于发货状态（卖家已发货/交易成功），以及是否全部行属于未发货状态（待审核/待打印快递单）。

### 问题 2：已执行工单重复进入待确认列表
**影响范围：2条差评**  
`fb-1777140947215` / `fb-1777141738642`

**根因：** `reprocessOne()` 无条件重置状态为 `pending`，scan-finalize 可能对同一工单第二次入队执行完整流水线。若第二次自动执行失败（因平台已退款），降级为 `simulated`，工单重新出现在待确认列表。

**典型案例：**
- `100001776913840889237`：17:24:07 首次自动退款成功，17:31:33 再次采集推理 → 自动执行失败 → 进入待确认
- `100001776739910505734`：同样模式

**修复方案：** 在 `reprocessOne()` 开头检查历史 sim 是否已有 `executedAt`；在 `processOne()` 自动执行前做同样检查，有记录则直接归档为 `done`。

### 问题 3：物流采集不完整导致"全部退回"假阳性
**影响范围：2条差评**  
`fb-1777109137784` / `fb-1777140724884`

**根因：** `logistics` 命令只从工单弹窗中读取可见包裹，对于含赠品的分包订单（主品1包 + 赠品2包），可能只读到1个包裹。若该包裹显示退回，`allPackagesReturned([1 package])` 返回 true → 系统 approve，实际还有2个赠品包裹未退回。

**典型案例：**
- `100001777101374886999`：ERP 有3行发货，鲸灵物流只采集到1个包裹，该包裹显示退回 → 错误 approve "全部包裹已退回"

**修复方案：** 在 flow-5.3 中将 ERP 发货行数与采集到的包裹数对比，若采集不完整则 escalate 人工核查，不冒险 approve。

## P1：流程准确性问题

### 问题 4：终态"用户已取消"未识别 → escalate 而非 skip
**影响范围：2条差评**  
`fb-1777099604107` / `fb-1777109171484`

**根因：** `infer.js` 终态检查列表中只有：`已退款/退款成功/已完成退款/已关闭/已撤销`，缺少"已取消"/"用户已取消"。这类工单是买家主动撤销申请，不需要任何处理，应自动归档。

**修复方案：** 在终态列表中添加 `'已取消', '用户已取消'`。

---

### 问题 5：等待重查条件过严 → 有时效工单被上报为人工
**影响范围：1条差评**  
`fb-1777141173211`

**根因：** 等待重查条件要求 `queueItem.deadlineAt` 和 `queueItem.hoursUntilNextScan` 都非 null 才判断 `safeToWait`。若 `deadlineAt` 为 null（解析失败），`safeToWait = null`，有充足时效（1天7小时）的工单被上报人工而非标记等待重查。

用户反馈："退货快递单存在但货物状态非「已收到退货」" 是无意义的 escalate 原因——该状态永远存在，因为 ERP 售后工单建立时就有记录。

**修复方案：** 在 `safeToWait === null` 时，fallback 使用 urgency 文本（如"1天7小时"→31小时），若 urgency 充足（>12小时）则允许等待重查。

### 问题 6：拦截流程未闭环
**影响范围：2条差评**  
`fb-1776915194516` / `fb-1777099922047`

**当前状态：** 仅退款-已发货的拒绝拦截场景，系统生成 reject 决策并在 warnings 中写"需创建快递拦截提醒"，但实际提醒只在用户确认执行后才创建。等待重查期间，人工不知道需要拦截。

**用户期望：** 
1. 时效充足时进入等待重查，但应同时创建 Mac 提醒告知"有X个快递需要拦截"（而非等执行时再提醒）
2. 区分三个快递单号中哪些已拦截成功、哪些未拦截，只对未拦截的发提醒

**风险评估：** 这是功能增强，不是 bug fix。需要拆分快递行动显示逻辑，复杂度高。列为 P1 后续优化。

### 问题 7：数量计算错误（商品档案 subItemNum 未正确计算赠品）
**影响范围：3条差评**  
`fb-1776934565784` / `fb-1777139696927` / `fb-1777141000907`

**核心问题：** `防晒*2支 赠防晒口罩*1` 类型的 SKU，商品档案中记录的子商品明细包含了赠品口罩，但 ERP 入库的口罩是赠品子订单而非主品子订单。导致计算"应退数量"时赠品被计入但实际入库记录里分开记录。

**典型案例：**
- `100001776839492218778`：sku=防晒*2支+赠口罩*1，subItemNum=3，approve 条件是 qtyGood(2) >= afterSaleNum(1)*subItemNum(3)=3 → 2<3 应 escalate，但实际 approve（说明 subItemNum 被计算为2）
- `100001776685503598817`：sku防晒*2支对应数量应为2，采集结果为1（shop-map 选了错误店铺）

**注意：** `100001776685503598817` 是 shop-map 问题（选错 ERP 店铺），`fb-1777142100990` 反映的 `100001776867996417927` 也是同一 shop-map 问题（共途账号被映射到上海绰绰店铺查询）。

**shop-map 问题已确认有修复路径：** shop-map.js 中 `共途` 关键词已存在，问题可能是 product-match 命令中店铺切换逻辑没有成功切换到正确店铺，而不是映射表本身有误。需进一步调查。

### 查看已退款工单 ERP 状态的问题（2条）
`fb-1777099349918` / `fb-1777139850473` / `fb-1777140420630`

工单已退款成功后，ERP 订单状态变为"已关闭"，系统无法获取到状态报"未获取到ERP状态"。用户期望：
