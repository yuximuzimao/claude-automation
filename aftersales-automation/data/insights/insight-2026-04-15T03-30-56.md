# 洞察报告 2026-04-15T03-30-56

## 推理规则分析与修改建议

### 问题1：拒收 — 缺少时间敏感决策 + 退回状态检测不可靠

**两条差评/好评共同指向同一场景**

当前逻辑对「已通知拦截、暂未退回」一律 reject，存在两个缺陷：

- **缺陷A**（FB1）：剩余时间充足时应先提醒拦截，等下次刷新再判断，而非直接拒绝
- **缺陷B**（FB3）：仅读工单详情的发货物流，但该字段存在不更新退回信息的情况，导致误判为「未退回」

**修改规则：**
```
拒收处理流程：
1. 双源核查退回状态：
   - 读工单详情发货物流
   - 读 ERP 发货物流
   - 任一显示退回 → 判定为已退回，走退款流程

2. 未退回时的时间分支：
   - 剩余处理时间 > 1天 → remind（提醒跟进拦截），不直接 reject
   - 剩余处理时间 ≤ 1天 且 未退回 → reject（备注等退回再退款）
```

### 问题2：重复快递单号检测 — 把当前工单自身计入「关联工单」

**FB2 的根本原因**：检测到工单号 100001775947313424249 与「已关联工单 100001775947313424249」冲突，实为同一工单，逻辑误将自身当成重复方。

**修改规则：**
```
检测退货快递单号是否被多单使用时：
- 从关联列表中剔除当前工单号本身
- 仅当存在【其他】工单号共用同一快递单号时，才触发 escalate
```

---

### 问题3：少件/漏发 — 未区分原单与补发单（ERP 子单格式）

**FB4 的根本原因**：ERP 返回的 `738694232-1` 是补发单（子单号格式 = 原单号 + `-N`），系统把其状态「待发货」当成无法识别的 ERP 状态报错。

**修改规则：**
```
少件/漏发场景 ERP 查询逻辑：
1. 识别补发单格式：平台交易号 = 原始单号 + "-1"/"-2" 后缀
2. 若 ERP 存在对应补发单（任意 -N 子单）：
   - 状态为「待发货」→ escalate，备注「已有补发单待发货，请核实后拒绝退款并填写补发快递单号」
   - 状态为「已发货」→ reject，备注补发快递单号
3. 若 ERP 不存在补发单 → 核实是否确实漏发，走正常漏发流程
```

### 问题4：质量问题 — 未识别「未拆封」作为超期退货特征

**FB5**：质量问题工单含「未拆封」字样时，实为超期退货场景，当前逻辑直接 escalate 走质量问题流程。

**修改规则：**
```
质量问题前置检查：
- 若工单描述/买家留言包含「未拆封」关键词
  → 重分类为超期退货场景，按超期退货规则处理
  → 不走质量问题流程
```

---

### 问题5：退货入库核验 — 未排除免退配件

**FB6**：3 种包装配件不需要退回入库，但系统计入应退数量，导致误判入库不足。

**修改规则：**
```
退货入库数量核验时，从发货明细中过滤以下配件（不计入应退数量）：
- HEE悦希雪梨纸（含「悦希雪梨纸」）
- HEE悦希印花礼袋-白（含「悦希印花礼袋」）
- HEE悦希印花礼盒（天地盖）白色（含「悦希印花礼盒」）

核验逻辑：
应退数量 = 发货明细总数 - 上述配件数量
实际入库 = 仓库登记数量（同样排除配件）
两者相符 → approve
```
