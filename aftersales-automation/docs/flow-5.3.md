# 仅退款-已发货处理流程（5.3，含拒收）

> 前置：已读 `docs/INDEX.md`，工单类型确认为「仅退款」，且 ERP 搜索显示已发货
> 核验重点：**所有发出包裹的物流节点，判断是否已退回或能否拦截**

---

## 完整流程

### Step 1：读鲸灵工单详情

```bash
node cli.js read-ticket <工单号>
```

记录：
- `subBizOrderDetailDTO[0].subBizOrderId` — 主子订单号
- `subBizOrderDetailDTO[0].afterSaleNum` — 买家申请退款套数
- `subBizOrderDetailDTO[0].spuBarcode` — 货号（套件判断时用）
- `giftSubBizOrderDetailDTO[0].subBizOrderId` — 赠品子订单号（如有）

### Step 2：ERP 获取完整发货包裹列表

> ⚠️ 以 ERP 行数为准，ERP 行数 > 鲸灵弹窗 tab 数时以 ERP 为准

```bash
# 搜主子订单号，记录所有行的快递单号 + 每行发货件数
node cli.js erp-search <主子订单号>

# 若有赠品
node cli.js erp-search <赠品子订单号>
```

### Step 2b：核验买家申请份数（⚠️ 多份发货必查）

> 需要先查档案V2的 subItemNum。如果不确定是否套件，先执行 `docs/erp-query.md` 中的商品查询。

```
ERP 发货总件数 = 所有行发货件数之和
档案V2.subItemNum = 套件单品数（单品时=0，按1计）
ERP 实际发货套数 = ERP发货总件数 ÷ subItemNum（单品时分母=1）
鲸灵 afterSaleNum = 买家申请退款套数

├─ ERP实际发货套数 = afterSaleNum → 正常，继续Step 3
└─ ERP实际发货套数 > afterSaleNum → ⚠️ 买家少申请了退款份数 → 上报人工
   备注格式：【待人工】买家实际购买X套（ERP发货X件），仅申请了Y套退款，
             建议确认买家是否只退Y套或需补申请
```

### Step 3：逐一查每个快递单号的物流最新节点

```bash
# 鲸灵物流（包含鲸灵弹窗各tab信息）
node cli.js logistics <工单号>

# ERP 物流（双源核查：鲸灵有时退回状态更新不及时）
node cli.js erp-logistics 0
```

**双源核查退回逻辑（满足任意一条即视为已退回）：**
- 鲸灵物流或 ERP 物流含以下任意关键词：退回商家 / 退回 / 安排退回 / 拒收 / 退件

> ⚠️ 鲸灵物流可能不更新退回状态，ERP 物流为补充核查来源，任意一源显示退回即可判定。

```
所有包裹均有退回物流（鲸灵或ERP任一源） → ✅ 同意退款（Step 4a）
任意包裹无退回物流 → 进入 Step 4（看无退回包裹的最新节点）
```

**特殊情况：部分退回 + 部分在途**
- 有包裹退回中 + 有包裹在途 → **整个工单全部拒绝**（不支持部分同意，现有工具无法拆分退款金额）
- 拒绝文案说明部分包裹状态，例如：`包裹A（YTxxxxxx）已退回，包裹B（YTyyyyyy）仍在途，请等所有包裹退回后再处理`
- 为在途包裹创建拦截提醒（Step 4c）

### Step 4：无退回物流的包裹——按最新节点判断

```
在途（快递在路上，买家未收）
  → 系统按「工单剩余时间 vs 距下次扫描时间」自动判断：
    ├─ 剩余时间 > 距下次扫描 → 自动标记「等待重查」（折叠，下次扫描自动重采集）
    └─ 剩余时间 ≤ 距下次扫描 → 拒绝退款（Step 4b），文案见下
  文案（拒绝时）：订单已发出，已通知快递拦截暂未退回，等快递退返回我司后再退款
  后续（拒绝时）：创建拦截提醒（Step 4c）

已签收（买家已实际签收）
  → 拒绝退款（Step 4b）
  文案：商品已签收，无法拦截，请自行申请退货退款
  后续：无需提醒
```

### Step 4a：同意退款

```bash
node cli.js approve <工单号>
```

### Step 4b：拒绝退款

⚠️ 截图凭证：截未退回包裹对应的物流截图（多包裹时需切换到对应tab再截图）

```bash
node cli.js reject <工单号> <原因> "<详情文案>" <图片URL>
```

### Step 4c：创建拦截提醒（在途/驿站未取件时）

```bash
# ⚠️ 不设日期——直接挂在提醒事项列表中
osascript << 'EOF'
tell application "Reminders"
  make new reminder at end of list "提醒事项" of default account with properties {
    name: "⚠️ 拦截快递：{快递单号}（{快递公司}）/ 子订单 {子订单号} / {账号} 工单{工单号}",
    body: "请操作快递拦截，退回后处理售后退款"
  }
end tell
EOF
```

## 多包裹特殊情况（5.3a）

**触发场景**：ERP 搜索同一子订单号返回多行，部分包裹已签收、部分包裹退回中。

**禁止直接按「有包裹签收→拒绝」处理**，必须先核查是否有历史退货：

```bash
# Step 1: 读鲸灵协商记录（从工单详情 innerText 搜「退货物流」「退回快递」）
node cli.js read-ticket <工单号>
# → 找买家在历史工单中填写的退货单号

# Step 2: ERP 订单管理搜子订单号，读订单行备注
node cli.js erp-search <子订单号>
# → 备注格式：[退货单号]; [规格编码]; *N（N=已入库件数）

# Step 3: ERP 售后工单新版搜协商记录中的退货单号
node cli.js erp-aftersale <退货单号>
# → 确认入库良品数量
```

计算：
```
已入库件数（Step 2/3）+ 退回中包裹件数（有退回节点）= 合计退回数
├─ 合计退回数 ≥ 应退件数 → 同意退款
└─ 合计退回数 < 应退件数 → 拒绝，说明缺少部分包裹
```

> 历史退货已入库但在当前工单 ERP 售后搜索中不出现 → 看 ERP 订单行备注的 *N 字段

## 决策快速参考

```
ERP获取所有发出包裹快递单号（主+赠品）
  ↓
核验买家申请套数 vs ERP实际发货套数（Step 2b）
  ↓
逐一查物流（鲸灵+ERP双源）
  ├─ 全部已退回（任意源） → 同意退款
  └─ 有未退回包裹
      ├─ 部分退回+部分在途 → 按在途处理（拒绝+创建拦截提醒）
      ├─ 在途 → 系统自动判断：剩余>下次扫描→等待重查；否则→拒绝+创建拦截提醒
      ├─ 驿站未取件（拒收）→ 拒绝+创建退回提醒
      └─ 已签收 → 拒绝+让改退货退款
  ↓（拒绝时）
截未退回包裹物流截图 → 拒绝退款操作
```
