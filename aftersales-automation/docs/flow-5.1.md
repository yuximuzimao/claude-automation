# 退货退款处理流程（5.1）

> 前置：已读 `docs/INDEX.md`，工单类型确认为「退货退款」
> 核验重点：**客户寄回的快递是否已在ERP入库，且入库明细和数量正确**

---

## 完整流程

### Step 1：读鲸灵工单详情

```bash
node cli.js read-ticket <工单号>
```

必须记录以下字段：
- `subBizOrderDetailDTO[0].subBizOrderId` — 主子订单号
- `subBizOrderDetailDTO[0].spuBarcode` — 货号
- `subBizOrderDetailDTO[0].attribute1` — SKU属性（attr1，用于商品对应表匹配）
- `subBizOrderDetailDTO[0].afterSaleNum` — 退货SKU件数（⚠️ 必读，影响应退数量计算）
- `giftSubBizOrderDetailDTO[0].subBizOrderId` — 赠品子订单号（如有）
- `giftSubBizOrderDetailDTO[0].spuBarcode` — 赠品货号（如有）
- 客户退货快递单号（buyerLogisticsNo）
- 售后说明（buyerRemark）+ 是否有售后图片

**afterSaleNum 计算规则（⚠️ 必须用，不能靠名字推断）：**
- 应退主商品单品数 = 档案V2.subItemNum × afterSaleNum
- afterSaleNum=2 → 退2件套装，应退单品数成倍增加
- 赠品数量始终1份，不随 afterSaleNum 倍增

### Step 2：查退货商品明细

> ⚠️ 必须查，不靠记忆或推算

**主商品**（参考 `docs/erp-query.md`）：

```bash
# 商品对应表：货号 → 规格商家编码
node cli.js product-match <货号> "<attr1>" <ERP店铺名>

# 商品档案V2：规格商家编码 → 单品名称+数量（subItemNum）
node cli.js product-archive <规格商家编码>
```

**赠品**（如有）：赠品货号走同样路径获取规格商家编码和单品信息。

### Step 2b：包裹级明细核验（数量不符时必做）

> ⚠️ 禁止只对比总件数得出"少退"结论——必须核验哪些包裹已签收、哪些未签收

```bash
# ERP 搜主子订单号，读每个包裹的快递单号和发货件数
node cli.js erp-search <主子订单号>

# 鲸灵查物流（读各包裹最新节点）
node cli.js logistics <工单号>
```

判断：
- 签收包裹的内容已退回入库 → 正常
- 未签收包裹（驿站/在途）→ 拦截件，等待退回，**不能视为"少退"**

### Step 3：ERP 售后工单新版搜退货快递单号

```bash
node cli.js erp-aftersale <退货快递单号>
```

展开所有「货物状态=卖家已收到退货」的行，逐行核对：
- 商品全称、数量、良品/次品（qtyBad 字段）

> ⚠️ **免退配件（不计入应退/实退数量）**：以下3种包装配件无需退回，ERP入库明细中若含这些商品需排除在数量核验之外：
> - HEE悦希雪梨纸（含「悦希雪梨纸」）
> - HEE悦希印花礼袋-白（含「悦希印花礼袋」）
> - HEE悦希印花礼盒（天地盖）白色（含「悦希印花礼盒」）

**搜不到退货单号时的处理**：
- 无售后说明 + 无图片，且剩余时效 - 距下次扫描 > 8h → 系统自动标记「等待重查」
- 无售后说明 + 无图片，且剩余时效 - 距下次扫描 ≤ 8h → 自动拒绝（剩余时效不足，需立即处理防止超时自动退款）
- 有售后说明或有图片 → 跳过自动等待，上报人工核查

### Step 4：验收判断

```
主商品数量吻合 + 全良品 → ✅ 继续判断赠品
主商品数量吻合 + 含次品 → ⚠️ 上报人工（见 INDEX §0.2）

有赠品：
├─ 赠品有入库记录 + 数量吻合 + 全良品 → ✅
├─ 赠品有入库记录 + 含次品 → ⚠️ 上报人工
└─ 赠品无入库记录 → 进入 Step 4b（赠品未退回）

主商品数量不符（实退 < 应退）：
├─ 先做包裹级核验（Step 2b），确认未签收包裹（在途）不算少退
├─ 排除在途因素后仍确认少退：
│   ├─ 鲸灵工单「售后总金额」= 原订单「支付金额」（买家按全额申请但只退了部分）
│   │   → 拒绝退款，要求买家按实退数量重新申请
│   │   文案："退回X个（购买Y个），请修改退款金额后重新申请"
│   └─ 鲸灵工单「售后总金额」< 原订单「支付金额」（买家已按比例申请）
│       → 核实比例：售后总金额 / 支付金额 是否约等于 实退件数 / 应退件数
│       ├─ 比例相符（误差≤5%）→ 同意退款
│       └─ 比例明显不符 → 上报人工判断
│   ⚠️ 无法自动计算或判断比例时，一律上报人工

全部通过 → Step 5 同意退款
```

### Step 4b：赠品未退回

```bash
# 搜赠品子订单号查发出快递物流
node cli.js erp-search <赠品子订单号>
```

```
有退回物流（退回记录存在）→ 等待入库，暂不处理，上报人工跟进
无退回物流 → 拒绝退款
  原因：赠品未退回
  文案：已收到退回主商品，但赠品暂未收到退回，请将赠品一并退回后再申请退款
  操作：见 Step 5（拒绝）
```

### Step 5a：同意退款

```bash
node cli.js approve <工单号>
```

### Step 5b：拒绝退款

需要物流截图作为凭证（以下步骤依次执行）：

1. 读物流文字（已在 Step 2b 中读取）
2. 截物流截图（需先在鲸灵弹窗切换到对应包裹 tab）
3. 执行拒绝命令：

```bash
node cli.js reject <工单号> <原因> "<详情文案>" <图片URL>
```

拒绝原因文案参考：
- 赠品未退回：`已收到退回主商品，但赠品暂未收到退回，请将赠品一并退回后再申请退款`
- 实退少退（未调整金额）：`退回X个（购买Y个），请修改退款金额后重新申请`

---

## 决策快速参考

```
鲸灵读详情（记录货号/赠品/退货快递单号/afterSaleNum/售后说明）
  ↓
商品对应表+档案V2 → 核实单品明细（必查）
  ↓
ERP售后工单新版 → 搜退货快递单号 → 展开所有入库行 → 核对名称/数量/良品
  ↓
全通过（无次品）→ 同意退款
次品 → 上报人工
赠品无入库 → 查赠品发出快递
  ├─ 有退回物流 → 等待入库，人工跟进
  └─ 无退回物流 → 拒绝退款（赠品未退回）
数量不符 → 先包裹级核验（未签收≠少退）→ 再判断金额是否按比例调整
```

## 已知坑位（退货退款流程）

- `[∞/永久保留]` **#6 必须展开所有行**：ERP 售后工单新版必须展开**所有**货物状态=已收到退货的行，不能只看第一行
- `[∞/永久保留]` **#12 多包裹快递单号不同**：物流弹窗每个 tab 是不同快递单号，禁止假设多包裹快递单号相同，必须逐 tab 读取
- `[∞/永久保留]` **#13 包裹数以ERP为准**：包裹数量必须以 ERP 订单管理搜索结果行数为准，鲸灵弹窗 tab 数可能少于实际包裹数
- `[∞/永久保留]` **#15 逐一核验再同意**：ERP 搜完后必须逐一核验所有包裹物流退回状态，不能搜完直接同意
- `[∞/永久保留]` **#31 "退未拆的"不违规**："退未拆的"= 只退未开封部分，符合规则；关键是退款金额是否按实退数量调整，若金额=全额则需拒绝要求买家修改
- `[∞/永久保留]` **#36 必读 afterSaleNum**：必须从 subBizOrderDetailDTO 读 `afterSaleNum`，不能靠 attr1 文字推断应退数量；应退主商品总数 = subItemNum × afterSaleNum；赠品不倍增
- `[∞/永久保留]` **#44 仅退款多包裹**：必须核对「ERP发货套数 vs 买家申请套数」，ERP发货总件数 ÷ subItemNum > afterSaleNum → 上报人工
- `[∞/永久保留]` **#45 数量不符先包裹核验**：数量不符时禁止直接判"少退"，必须先做包裹级核验区分在途拦截件和真实少退
